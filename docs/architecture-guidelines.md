# Architecture Guidelines

## HCC Micro-Frontend Architecture

Scheduler UI is a **micro-frontend** in the Hybrid Cloud Console (HCC) ecosystem. It runs inside the [insights-chrome](https://github.com/RedHatInsights/insights-chrome) shell application, which provides:

- Navigation (left sidebar, breadcrumbs)
- Authentication and identity management
- Notifications system
- Help panel and learning resources
- Module federation runtime (loading federated modules)

## Module Federation

### Configuration (`fec.config.js`)

The `fec` CLI (from `@redhat-cloud-services/frontend-components-config`) wraps webpack with HCC-specific defaults:

```javascript
module.exports = {
  appUrl: '/apps/scheduler-ui',        // Route where the app is mounted
  appEntry: './src/AppEntry.js',        // Webpack entry point
  moduleFederation: {
    exposes: {
      './RootApp': './src/AppEntry',    // Federated module exposed to the shell
    },
    exclude: ['react-router-dom'],      // Provided by the shell at runtime
    shared: [{ 'react-router-dom': { singleton: true, import: false } }],
  },
};
```

### Entry Point Flow

1. **insights-chrome** discovers the app via the Frontend CRD manifest
2. Chrome loads `/apps/scheduler-ui/fed-mods.json` (generated by `fec build`)
3. Chrome imports the `./RootApp` remote module
4. `AppEntry.js` renders `<BrowserRouter><App /></BrowserRouter>`
5. `App.js` defines routes using React Router v6

### Important Constraints

- **react-router-dom is shared**: The shell provides it. Don't add it to `dependencies` with a conflicting version â€” the `shared` config ensures singleton behavior.
- **No standalone mode**: The app cannot run independently in production. The dev server (`fec dev`) proxies chrome for local development.
- **CSS isolation**: Use the `sassPrefix` (`.scheduler-ui`) to scope all styles. Global styles will bleed into other apps.

## Frontend CRD (`deploy/frontend.yaml`)

The `Frontend` Custom Resource tells the [frontend-operator](https://github.com/RedHatInsights/frontend-operator) how to deploy and register this app in OpenShift:

```yaml
spec:
  title: "Report Scheduler"
  frontend:
    paths:
      - "/apps/scheduler-ui"
  module:
    manifestLocation: "/apps/scheduler-ui/fed-mods.json"
    modules:
      - id: scheduler-ui
        module: "./RootApp"
```

Key fields:
- `paths`: URL paths this app serves
- `manifestLocation`: Where chrome finds the module federation manifest
- `feoConfigEnabled`: Enables Frontend Experience Orchestrator integration

## Adding New Features

### New Page/Route

1. Create component in `src/Components/NewPage.js`
2. Add route in `src/App.js`:
   ```javascript
   <Route path="/new-page" element={<NewPage />} />
   ```
3. Add unit test as `src/Components/NewPage.test.js`

### New Shared Component

1. Create in `src/Components/` with PascalCase naming
2. Import PatternFly components from `@patternfly/react-core`
3. Co-locate test file
4. Use PF5 CSS variables for any custom styling

### API Integration

When adding API calls:

1. Use `@redhat-cloud-services/frontend-components-utilities` for auth-aware fetch
2. Chrome provides the authentication token automatically via the proxy
3. API base URLs are resolved through the HCC proxy in dev and the gateway in production

## CI/CD (Tekton / Konflux)

### Pull Request Pipeline (`.tekton/scheduler-ui-pull-request.yaml`)

Runs on every PR:
- Build the app
- Run unit tests
- Run E2E tests against the built app served by Caddy
- Container image build and security scan

### Push Pipeline (`.tekton/scheduler-ui-push.yaml`)

Runs on merge to `master`:
- Full build + test
- Container image push to Quay
- Deployment to stage environment

### Legacy PR Check (`pr_check.sh`)

Uses `insights-frontend-builder-common` for building. This is the legacy Jenkins-based build â€” Tekton/Konflux pipelines are the primary CI system.

## Static Asset Serving (Caddy)

The `Caddyfile` configures how built assets are served:

- Listens on port 8000
- Strips `/apps/scheduler-ui` prefix from requests
- Serves files from `dist/`
- Falls back to `index.html` for client-side routing
- Provides `/health` endpoint for readiness/liveness probes
